## R1: Facebook Messenger Export Format

### Export Formats

Facebook's "Download Your Information" (facebook.com/dyi) produces two formats:

- **JSON**: Structured data with `participants`, `messages`, `title`, `thread_path`. Split into files — `message_1.json` (newest), `message_2.json` (older), etc.
- **HTML**: Human-readable format with CSS-classed DOM structure. One or more `.html` files per conversation.

### JSON Structure

```json
{
  "participants": [{ "name": "Nguy\u00e1\u00bb\u0085n V\u00c4\u0083n A" }],
  "messages": [
    {
      "sender_name": "Nguy\u00e1\u00bb\u0085n V\u00c4\u0083n A",
      "timestamp_ms": 1708000000000,
      "content": "Xin ch\u00c3\u00a0o",
      "type": "Generic",
      "is_unsent": false
    }
  ],
  "title": "Nguy\u00e1\u00bb\u0085n V\u00c4\u0083n A",
  "thread_path": "inbox/NguyenVanA_abc123"
}
```

### Mojibake Encoding Problem

Facebook JSON exports encode Vietnamese text as Latin-1 byte escapes instead of proper UTF-8. The string `"Xin chào"` appears as `"Xin ch\u00c3\u00a0o"` in the JSON.

**Fix**: Decode each byte back through UTF-8:

```typescript
function decodeFBString(str: string): string {
  try {
    return new TextDecoder("utf-8").decode(new Uint8Array([...str].map((c) => c.charCodeAt(0))));
  } catch {
    return str; // If not mojibake, return original
  }
}
```

Apply to ALL string fields: `sender_name`, `content`, `title`, participant `name`.

### HTML Structure

HTML exports use obfuscated CSS classes (likely generated by React):

- Conversation title: `._a6-h._a6-i`
- Participants: `._a6-p`
- Message block: `.pam._3-95._2ph-._a6-g`
  - Sender: first `._3-94._2lem` child
  - Content: `._2ph_._a6-p > div`
  - Timestamp: last `._3-94._2lem` child
  - Reactions: `ul._a6-q > li`

HTML is properly UTF-8 encoded — no mojibake fix needed.

### Key Observations

- **Message order**: Both formats store messages newest-first → must reverse for chronological display
- **Multi-file merging**: `message_2.json` contains older messages than `message_1.json` → merge and sort by timestamp
- **Message types**: `"Generic"` = text, `"Share"` = link, photos/videos/stickers have separate fields
- **Unsent messages**: `is_unsent: true` — content is null, show placeholder

## R2: Google Gemini SDK

### SDK Choice

| Package                 | Status            | Notes                               |
| ----------------------- | ----------------- | ----------------------------------- |
| `@google/genai` v1.x    | **GA (use this)** | Current official SDK, released 2025 |
| `@google/generative-ai` | **DEPRECATED**    | EOL August 2025. Do NOT use.        |

### Basic Usage

```typescript
import { GoogleGenAI, Type } from "@google/genai";

const ai = new GoogleGenAI({ apiKey: userApiKey });

const response = await ai.models.generateContent({
  model: "gemini-2.5-flash",
  contents: [{ role: "user", parts: [{ text: systemPrompt + "\n\n" + JSON.stringify(batch) }] }],
  config: {
    responseMimeType: "application/json",
    responseSchema: {
      type: Type.ARRAY,
      items: {
        type: Type.OBJECT,
        properties: {
          id: { type: Type.NUMBER },
          score: { type: Type.NUMBER },
        },
        required: ["id", "score"],
      },
    },
    thinkingConfig: { thinkingBudget: 0 },
  },
});

const scores = JSON.parse(response.text);
```

### Key Decisions

- **Model**: `gemini-2.5-flash` — fast, cheap, good at structured output
- **Structured output**: `responseMimeType: "application/json"` + `responseSchema` forces Gemini to return valid JSON matching the schema. No JSON parsing errors.
- **Thinking budget**: `thinkingBudget: 0` disables "thinking" tokens → 3-5x cost reduction for classification tasks that don't need chain-of-thought
- **Browser-compatible**: SDK works directly from browser (`fetch` to `generativelanguage.googleapis.com`)

### Rate Limits (Free Tier)

| Limit               | Value         |
| ------------------- | ------------- |
| Requests per minute | 15 RPM        |
| Tokens per minute   | 1,000,000 TPM |
| Requests per day    | 1,500 RPD     |

For 5,000 messages (50 batches at 100/batch): ~50 requests × 4s delay = **~3.5 minutes**.

## R3: react-virtuoso

### Why react-virtuoso

- **Bundle size**: ~25KB gzip (acceptable for our <2MB budget)
- **Variable height items**: Auto-measures item heights — essential for chat bubbles with varying text length
- **GroupedVirtuoso**: Built-in support for grouped lists with sticky group headers → perfect for date separators
- **Scroll-to-index**: Programmatic scrolling API for search navigation
- **followOutput**: Auto-scroll to bottom when new items arrive

### Usage Pattern

```tsx
import { GroupedVirtuoso } from "react-virtuoso";

<GroupedVirtuoso
  groupCounts={groupCounts} // [5, 12, 8] — messages per date group
  groupContent={(
    index, // Date separator header
  ) => <DateSeparator date={groupLabels[index]} />}
  itemContent={(
    index, // Individual chat bubble
  ) => <ChatBubble message={flatMessages[index]} />}
  defaultItemHeight={72} // Estimated height for unmeasured items
  followOutput="smooth" // Auto-scroll on new items
/>;
```

### Performance

- Renders only visible items + small overscan buffer
- 60fps scrolling verified with 50K+ items
- Auto-measurement eliminates need for manual height calculation
- `defaultItemHeight` reduces layout shifts during fast scrolling

### Alternatives Considered

| Library           | Size  | Variable Height | Groups             | Decision                           |
| ----------------- | ----- | --------------- | ------------------ | ---------------------------------- |
| react-virtuoso    | ~25KB | ✅ Auto         | ✅ GroupedVirtuoso | **Selected**                       |
| @tanstack/virtual | ~5KB  | ✅ Manual       | ❌ Manual          | Too much boilerplate for groups    |
| react-window      | ~6KB  | ❌ Fixed only   | ❌ No              | Can't handle variable chat bubbles |

## R4: Keyword Search

### MVP Approach

Simple client-side text search — no external library needed.

```typescript
function searchMessages(messages: Message[], query: string): Message[] {
  if (!query.trim()) return messages;
  const lower = query.toLowerCase();
  return messages.filter((m) => m.content?.toLowerCase().includes(lower));
}
```

### Highlighting

Split matched text for `<mark>` wrapping:

```typescript
function highlightMatch(text: string, query: string): (string | JSX.Element)[] {
  if (!query) return [text];
  const regex = new RegExp(`(${escapeRegex(query)})`, 'gi');
  return text.split(regex).map((part, i) =>
    regex.test(part) ? <mark key={i}>{part}</mark> : part
  );
}
```

### Design Decisions

- **Case-insensitive**: Always lowercase comparison
- **No fuzzy matching**: MVP uses exact `String.includes()`. Fuzzy search (e.g., fuse.js) deferred to post-MVP
- **No regex**: Users enter plain text queries only
- **No date-range filter**: Deferred to future iteration
- **Debounced input**: 150ms debounce to avoid filtering on every keystroke
- **Navigation**: ↑↓ buttons to jump between matches, integrated with Virtuoso `scrollToIndex`

## R5: Dexie.js v4 (IndexedDB)

### Why Dexie.js

- **Bundle size**: ~20KB gzip
- **TypeScript-first**: Full type safety with generics
- **Compound indexes**: `[conversationId+order]` for efficient range queries
- **Bulk operations**: `bulkPut()` for fast batch inserts (thousands of messages)
- **React integration**: `dexie-react-hooks` provides `useLiveQuery()` for reactive queries

### Schema Design

```typescript
import Dexie, { Table } from "dexie";

class SenMessageDB extends Dexie {
  conversations!: Table<Conversation>;
  messages!: Table<Message>;
  batches!: Table<AnalysisBatch>;

  constructor() {
    super("SenMessageDB");
    this.version(1).stores({
      conversations: "id, title, messageCount",
      messages: "id, [conversationId+order], [conversationId+sweetnessScore], conversationId",
      batches: "++id, conversationId, status",
    });
  }
}
```

### Query Patterns

```typescript
// All messages for a conversation, chronological
db.messages.where("[conversationId+order]").between([convId, Dexie.minKey], [convId, Dexie.maxKey]).toArray();

// Sweet messages at level 3+
db.messages.where("[conversationId+sweetnessScore]").between([convId, 3], [convId, 5]).toArray();

// Bulk score update after AI batch
db.messages.bulkPut(scoredMessages);
```

### Reactive Updates with useLiveQuery

```typescript
import { useLiveQuery } from "dexie-react-hooks";

const messages = useLiveQuery(() => db.messages.where({ conversationId: convId }).sortBy("order"), [convId]);
```

Changes to IndexedDB data automatically trigger React re-renders. Used for progressive AI results — as scores arrive, the UI updates without manual state management.

### Performance Notes

- `bulkPut()` with 5,000 messages: < 500ms
- Compound index queries: < 10ms for 50K rows
- In-memory `Array.filter()` for slider: < 5ms for 50K items (faster than IndexedDB query per slider change)
- **Strategy**: Load all conversation messages into memory once, then filter in-memory for slider changes. Only use IndexedDB for persistence and initial load.

### Messenger-Style Chat Bubbles

```html

    <p class="text-sm">Nhớ em quá đi</p>
    <span class="text-xs opacity-70">10:30</span>
  </div>
</div>

<!-- Other's message (left-aligned, gray) -->
<div class="flex justify-start">
  <div class="max-w-[85%] sm:max-w-[75%] bg-gray-100 text-gray-900 rounded-2xl rounded-bl-md px-4 py-2">
    <p class="text-xs font-medium text-gray-500 mb-1">Nguyễn Văn A</p>
    <p class="text-sm">Em cũng nhớ anh</p>
    <span class="text-xs text-gray-400">10:31</span>
  </div>
</div>
```

### Key Styling Decisions

- **Rounded corners**: `rounded-2xl` with directional override (`rounded-br-md` for own, `rounded-bl-md` for other) — mimics Messenger bubble shape
- **Max width**: `max-w-[85%]` mobile, `max-w-[75%]` desktop — prevents bubbles from spanning full width
- **Colors**: Blue-500 for own messages (Messenger signature blue), gray-100 for others
- **Responsive**: Mobile-first design, readable at 360px width
- **Date separator**: Centered text with horizontal divider lines

### Sweetness Indicator
